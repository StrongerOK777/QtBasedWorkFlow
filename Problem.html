节点式图像处理工作流工具

一、题目描述

爱乐普第橙汁公司的图像分析部门每天要处理海量的橙子图片。他们有时要把橙子从背景中抠出来做透明底，有时要分析橙子的色泽分布，有时要把一批橙子照片裁成统一尺寸做质检比对，有时要给客户发去的橙子图片加上公司logo水印，有时要调亮某张打光不好的橙子照，有时要把一张橙子图切成九宫格发社交媒体，有时甚至要把几张不同角度拍的橙子拼成一张全景图……

然而，每个人的处理流程都不一样：张三是先裁切→调亮度→加蒙版→导出，李四要先分离RGB通道→只保留R通道→增强对比度→再合并回去，王五则要先模糊背景→抠出橙子→风格化成“油画效果”→加文字标签→导出成某种尺寸。每次都手工操作或者临时写脚本，效率极低、容易出错，而且想法越多流程越乱。

部门负责人LAM提出：节点式图像处理工作流工具 is all you need，让每个人都像搭积木一样，把不同的操作节点拖出来、连上线、调好参数，一键执行整个流程。这个工具需要稳定、可扩展，还要能把搭建好的流程保存下来，下次直接回放，甚至用命令行批处理几百张橙子图。

你的任务就是：设计并实现这个工具的桌面原型。

你可以尽情发挥想象力：除了基础的裁切、调色、模糊等操作之外，是不是可以做个“橙子风格化”节点，把照片变成油画、素描或卡通风格？要不要连上外部API，支持“输入一段文字描述，生成一张橙子创意图片”作为输入节点？要不要做个“橙子品质评分”节点，输出一份简单的数值报告？你设计的不仅是工具，更是一个平台——未来可以不断扩展新的节点类型。当然，这些是启发，不是必做项，基础功能扎实更重要。

二、基本任务

你需要使用C++和Qt完成一个节点式图像处理工具，允许使用智能编程工具（如大模型）辅助开发。

你需要自己决定并完成的包括但不限于：

  1. 支持哪些图像处理节点

    • 必须覆盖基本的读图、导出、显示，以及几种常见的图像变换/滤波/色彩操作；你可以加入相对复杂的处理（如通道拆分合并、蒙版混合等），以及一些有创意的节点类型（风格化、文字叠加、拼图、调用外部API等，任你想象）。
    • 具体节点类型、数量、参数与算法由你设计，至少实现8种节点种类，各个节点的具体实现方法允许且建议你调库实现。
    • 不一定要做橙子相关，本次作业鼓励你做出自己的风格！你可以自己设计好玩的工作流，自己设计有趣的主题，只要涉及图像处理即可。

  2. 节点工具的设计

    • 每个节点的输入输出端口数量、数据类型规范（如彩色图、灰度图、图像列表等）
    • 端口兼容性检查规则（保证只有类型匹配的端口才能相连）
    • 节点间数据传递方式
    • 工作流必须为有向无环图，支持分支与汇合（例如一图分多路处理再合并）

  3. 图形界面布局 - 你需要一个基于Qt的图形界面，让用户可以：

    • 从列表中添加节点到编辑区
    • 在画布上移动节点、连接端口、删除节点或连线
    • 选中节点后修改其参数
    • 预览某一节点的输出图像
    • 一键执行整个流程，并多次重建、执行不同流程

界面布局、交互细节由你设计，要求清晰可用。

  1. 健壮性

    • 当参数错误、文件缺失或端口类型不匹配时，程序必须弹出清晰的警告或错误信息，不允许崩溃
    • 鼓励排查并处理常见异常

  2. 工作流持久化与命令行执行

设计一种文件格式（推荐JSON）来保存和加载工作流。若实现，还需支持命令行模式：

ImageNodeEditor --workflow workflow.json --no-gui

在此模式下不启动图形界面，执行完成后自动退出并输出成功或失败信息。

三、分析文档

在动手编写（或生成）代码之前，你必须先形成一份分析文档。这份报告旨在让你在编码前理清全部设计，同时鼓励你主动与AI工具对话以提升方案质量。

建议你根据下面步骤进行完成：

  1. 初稿构思 先根据本题背景，自行思考并写下初步的工具设计方案（大纲、草图均可）。

  2. AI交互优化

使用大语言模型或同类工具，把你的初步方案抛给它，与它讨论：

  • 节点类型如何设计更合理？
  • 端口数据类型规范有什么遗漏？
  • 界面布局是否便于使用？
  • JSON工作流格式该如何定义？
  • 可能出现哪些异常情况？
  • 其他问题。

根据讨论结果修改完善你的方案。

  1. 最终分析文档提交

最终提交的分析文档中，内容应至少包含：

  • 你对题目的理解与你的设计目标
  • 你设计的完整节点列表、每个节点的功能、输入输出端口与参数
  • 端口数据类型规范与兼容性矩阵
  • 图形界面布局设计（可附草图或线框图）
  • 工作流文件格式设计（如JSON示例）
  • 整体架构设计：模块划分、类层次结构示意图、数据流说明
  • 异常处理策略
  • AI工具使用说明：你使用了哪些提示词或对话过程，AI对方案的哪些部分提出了建议，你采纳或修改了什么，并简要说明理由。分析文档本身可以由AI辅助撰写或润色，但你必须能解释其中的每一个设计决策。

四、代码要求

无论你如何设计功能，你的代码必须满足以下要求：

  • 至少将图形界面、节点逻辑、图像处理算法、文件读写分离到不同的类/模块中。禁止将所有代码堆砌在主窗口类或单个.cpp文件中。

  • 必须通过继承、多态等方式抽象节点，严禁使用超长if-else或switch按节点类型穷举逻辑，如下所示：

    // 这是一个反面例子！
if (type == "crop") { ... }
else if (type == "blur") { ... }
else if (type == "brightness") { ... }
else if (type == "contrast") { ... }

  • 内部可以使用QImage、cv::Mat或自定义结构，但工作流层面的数据类型规范应明确，并保证连接时的类型安全。

  • 输入输出路径应通过文件对话框、参数或命令行参数指定，不允许写死！程序素材的读取路径（如有）应当使用相对路径，不允许使用绝对路径！

  • 尽量使用智能指针或清晰的所有权模型，避免内存泄漏和悬挂指针。

  • 如果同时实现GUI与命令行模式，二者应共享同一套图像处理与工作流执行逻辑，只是前端不同。
