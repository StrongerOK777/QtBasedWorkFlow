项目简介 · 技术栈
项目题目简介 // ABSTRACT
ImageNodeEditor 是一款桌面端节点式图像处理工作流工具。它把读图、裁切、缩放、调色、滤波、文字叠加、预览、导出等操作抽象为可视化「节点」——在画布上拖出节点、连线、改参数，即可像搭积木一样拼出一条图像处理流程，一键执行并实时预览，流程可保存为 JSON 复用。内置 27 种处理节点 + 1 个宏节点，覆盖输入输出、几何变换、色彩、滤波、合成、批处理六大类。图形界面（GUI）与命令行（picdeal）共用同一套核心引擎，行为完全一致：既能交互式探索，也支持无界面批处理。

Core Architecture
设计思路 · 分层解耦
设计思路：从第一天就坚持分层解耦——界面 / 数据模型 / 图像算法 / 执行引擎四层分离，每层可独立理解与测试，新增节点无需改界面代码；模块间以 Qt 信号槽 通信，整体借鉴 VS Code 的工作台分区。
界面层gui/ + qml/UI LAYER
QML 工作台MainWindowWorkflowCanvas · QtNodes参数面板预览对比对话框
▼
核心层workflow/ + nodes/CORE LAYER
图结构校验拓扑执行JSON 序列化ImageNode 多态NodeFactory
▼
基础层processing/ + core/ + util/BASE LAYER
纯图像算法端口 / 连线 / 参数类型结果封装路径工具
多态 + 工厂注册 // NO if-else
所有节点继承抽象基类 ImageNode；新增节点只需在 NodeFactory 注册一行，执行引擎零改动——彻底消除大量 if-else 分支。
class ImageNode { // abstract base virtual ports(); virtual execute(); }; NodeFactory::reg("Blur", …); // +1 line ⟶ done
引擎独立 · GUI / CLI 共用 // 1 ENGINE
GUI 的「执行」与命令行 picdeal 跑的是同一套 ExecutionEngine，行为一致；执行前统一校验，再按依赖拓扑执行，节点级缓存避免重复计算。
校验 端口/类型/DAG→ 拓扑排序→ 按序执行→ LRU 缓存→ 预览 / 导出

01退出时偶发崩溃：析构中信号回调访问到已半析构的对象
→
析构开头先断开信号连接，并加「正在关闭」标志位兜底
02工作台侧栏一片黑：windeployqt 默认不部署 QML 模块
→
windeployqt 加 --qmldir 指向 QML 源码目录，部署 Qt Quick
03命令行一启动就崩：缺 qoffscreen 离屏渲染平台插件
→
构建后自动复制 qoffscreen.dll 到 platforms/
04大图模糊很慢：逐像素取邻域均值，半径一大就卡顿
→
改「滑动窗口」均值算法，复杂度与半径无关 + LRU 缓存
05既交 VS 工程又迁移 macOS：两套构建体系要求冲突
→
双构建并存：原生 .slnx/.vcxproj + CMakeLists，同一份源码

心得体会 // CONCLUSION
最深的两点体会。其一，分层与多态真的很值：开始就把界面、模型、算法、执行分开，加节点基本只写一个算法函数再注册一行，后期扩到 27 个节点也不乱；若全堆在一个文件里早就改不动了。其二，跨平台与部署的坑比算法本身更费时间：真正反复折腾的不是图像处理，而是打包缺插件、QML 黑屏、构建配置这些环境问题——这让我体会到「能跑」和「在别人机器上也能跑」完全是两回事。
